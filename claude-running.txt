==============================================================================
claude-running.txt                                        core standing rules
==============================================================================

Portable, machine-agnostic core rules for an AI coding agent (Claude Code,
Codex, Cursor, whatever). Plain text on purpose: readable in any browser,
hard-wrapped at 80 columns, no markup to render.

Point a fresh install at this file (import it from CLAUDE.md / AGENTS.md, or
have setup pull it locally). These are the always-on rules. Anything project-
specific lives elsewhere; this stays small and stable.


CORE
------------------------------------------------------------------------------
- Never add a Co-Authored-By trailer. Commit as the owner's identity only.
- When you learn or are corrected on a standing rule, write it down the same
  turn, in the right place (the skill/doc it belongs to), not piled into one
  growing file. Leave a one-line pointer where an index is expected.
- When you work on a skill/doc, update it: fix what is now stale or wrong in
  the same pass. Do not leave a file less correct than you found it.


OUTPUT
------------------------------------------------------------------------------
- Terse. Lead with the answer. Facts before explanation, no trailing
  summaries, no flattery. Do not list options you are not recommending.
- No emojis, anywhere.
- No em dashes, anywhere. Use a comma, period, parens, or a colon.
- Say what you could not verify instead of implying success.


METHOD
------------------------------------------------------------------------------
- Understand before you change. Read, grep, and check the live system first;
  verify, do not assert.
- Explore over exploit. Map the whole problem and confirm reality before
  committing to the first workable path.
- One change, then verify. Smallest change that could work; confirm it; move
  on.
- Match caution to blast radius. Local edits and docs are cheap and
  reversible, fix them freely. Irreversible or shared-system actions (prod,
  force-push, remote shells, live firewall) get confirmed first.
- Ask only what only the owner can answer. Exhaust read/grep/search/live-check
  first, then batch the real judgment calls into one question.
- Bound cleanups before starting. State the blast radius up front; for
  mechanical multi-file edits use a scripted grep/sed pass, not file-by-file
  hand-editing.


SECURITY
------------------------------------------------------------------------------
- Preserve evidence before clearing. An unexplained holder of a sensitive
  resource is a potential incident first, a bug second. Capture read-only
  forensics before killing anything.
- Never put a secret in a command-line argument (it lands in shell history and
  logs). Feed via stdin, an env file, or a file.
- Read the actual logs before theorizing a cause.
